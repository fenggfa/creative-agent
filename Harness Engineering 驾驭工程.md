# Harness 工程：让系统更会纠错
## Harness = Agent 的控制系统

1. **目标状态**：定义什么叫“对”、什么叫“干净”。
2. **传感器**：持续感知偏差（测试、日志、规则）。
3. **执行器**：自动动作（改代码、回滚、重试）。
4. **核心机制 反馈回路**：保证每一步动作都在向目标收敛，而非无序发散。

> Harness 工程不是教你把 Agent 训成“乖宝宝”，而是让整个系统更会纠错。

## 三层进化链：从“表达”到“控制”的范式转移

| 局限 | 无法感知系统状态 | 仅限单次交互，无记忆性 |

## 维度打击：为什么 Harness 不是 Prompt 2.0
> “Prompt 是沟通，Harness 是管理。一个是教员工怎么说话，一个是给员工定 KPI、装监控、设审计流程。”

| 对比维度 | Prompt Engineering | Harness Engineering |
|---------|--------------------|---------------------|
| 管理对象 | 单次模型调用 | 整个 Agent 系统的演化 |
| 核心问题 | 这次怎么问更清楚？ | 系统偏了谁发现？谁修？ |
| 时间尺度 | 毫秒级 / 秒级 | 小时级 / 天级（长任务） |
| 调优单位 | 提示词模板 | 反馈回路、规则、约束 |

**核心结论**：Harness 是**系统级控制设计**，而非“更长的系统提示”。

## 软件工程的控制论：Kubernetes Controller
声明期望状态，控制器持续监控真实状态并自动补齐、修复、回滚

**K8s控制回路拆解**

## 核心组件拆解：拆解 Agent 的 "操作系统"
> 模型是 CPU，算力再强，没有操作系统也跑不出生产力。这四块凑齐了，才叫工程化。

### 目标状态怎么写才算“可执行”
把“脑子里的感觉”变成“仓库里的标准”

- **完成 (Functional)**：feature_list.json  
  明确界定哪些端到端行为必须成立，消除模糊地带。
- **可接受 (Quality)**：启动 < 800ms  
  设定不可妥协的性能、稳定性与可观测性门槛。
- **符合架构 (Arch)**：ARCHITECTURE.md  
  严格约束依赖方向、代码分层与模块边界。
- **改动干净 (Clean)**：QUALITY_SCORE.md  
  确保不制造新的技术债务，且变更随时可安全回滚。

> “别写‘请写得优雅’，那没用。凡是没写进仓库的规则，对Agent来说一律不存在。”

## 反馈回路：Agent摆脱盲飞的唯一解
从“盲飞”到“自校准”的闭环

1. **执行**：Agent基于上下文生成改动
2. **感知**：自动触发传感器（测试/日志）

5. **修正**：调用执行器进行最小范围修复
6. **再验证**：重复循环直到系统状态纯净

> 反馈回路是对抗熵增、实现长任务收敛的唯一手段。

## 熵管理：AI时代的垃圾回收（GC）
### 用“黄金原则”与后台清理任务持续纠偏，防止坏模式扩散

**黄金原则 Golden Principles**
1. **Parse, don't validate**：在边界处强制转换数据形状，拒绝隐式信任大模型的输出结构。
2. **DRY (Don't Repeat Yourself)**：严禁生成重复的 Helper 函数，强制复用共享工具链，避免逻辑碎片化。
3. **Explicit over Implicit**：所有约定必须显式编码，压缩 AI 的幻觉空间与猜测成本。

**持续清理流程**
- 扫描：后台 Agent 定期检索违背原则的代码模式。
- 重构：自动提交小颗粒度 Refactor PR。
- 合并：人类进行秒级 Review 并自动合并。

## 长任务为什么总烂尾？
### 长任务失败的结构性根因

- **换班失忆 (CONTEXT FRAGMENTATION)**  
  每个 Session 都是“纯净脑”，缺乏决策路径的连续性。压缩后的上下文彻底丢失了关键的“为什么这么做”。

- **一次性幻觉 (ONE-SHOT FALLACY)**  
  试图在单次调用中完成复杂功能，贪多嚼不烂。  
  导致：代码未测试、文档缺失、中间状态极其混乱。

**致命后果**：下一轮 Agent 需要花费高达 **80% 的 Token** 去“猜历史”，直接导致整个长任务的执行效率呈指数级下降，最终走向崩溃。

---

## 拒绝废墟盖楼：Agent时代的工程范式
### 两段式任务编排架构

**先遣队 Initializer Agent**
- 核心动作：生成项目骨架、编写 init.sh、初始化进度文件。
- 关键产出：结构化的环境和任务起点。

**主力部队 Coding Agent**
- 执行逻辑：每次只领一个任务，严格遵循实现 → 测试 → 提交循环。
- 绝对约束：只做增量，保持可验证。

**核心价值**：将长任务拆解为一系列可预测、可验证的增量步骤，让每一班 Agent 都在清爽的环境下接手。

---

## 拒绝 Agent 的「纸面胜利」
### 结构化完成标准 (Definition of Done)

feature_list.json 示例：
```json
{
  "id": "feat-001",
  "desc": "用户可创建新会话并收到回复",
  "steps": ["点击新会话", "输入文本", "验证回复"],
  "passes": false
}
```

- **剥夺篡改权限**：Agent 仅拥有只读权限，严禁修改 desc 与 steps 字段，从根本上杜绝“偷偷改需求”的可能。
- **唯一状态变更权**：将状态变更权死死绑定在测试链路上。仅在端到端 (E2E) 测试真实跑通后，才允许将状态翻转为 true。

> 这是硬指标：不准删需求，不准改描述。彻底终结 Agent 伪造测试结果的幻觉。

---

## 环境自愈与心跳检查
> 环境坏了，代码写得再好也没用。绝不在废墟上盖大楼。

**核心职责**
- 自动安装与更新依赖库，抹平环境差异
- 静默启动本地开发服务器与数据库
- 运行基础 Sanity Test（冒烟测试）

**工程博弈策略**
- **失败优先处理**：若脚本执行失败，Agent必须停止新功能开发，强制转入环境修复模式
- **Token护城河**：避免 Agent 在每个 Session 中重复探索“如何运行项目”，极大节省上下文开销。

---

## 打破 Agent 的“纸面胜利”
### 终极传感器：端到端（E2E）闭环

集成 Puppeteer / Playwright MCP 工具，强制 Agent 站在用户视角评估交付质量，精准识别“代码逻辑正确但 UI 无法交互”的隐蔽 Bug。

- **环境唤醒**：自动打开浏览器，建立交互基座
- **拟人交互**：模拟真实用户点击，触发状态流转
- **视觉断言**：捕获反馈，对比预期 UI 与真实 DOM

> “没有 E2E 验证的 Harness，充其量只是个半成品。”

## 降维打击：从代码防御到架构自治的闭环进化
